# AI 对话写文章 - 完整流程路径

## 概述

当用户在 AI 对话中要求写一篇文章时，系统会经历以下完整流程：

```
用户输入 → 前端处理 → 后端 API → 字数提取 → LLM 调用 → 流式输出 → 字数检查 → 自动续写（如需要）→ 完成
```

## 关键节点说明

### 1. 字数要求提取（第 4 步）
- **时机**：在第一次调用 LLM 之前
- **位置**：`_chat_generator` 函数开始时
- **作用**：决定是否启用智能字数控制

### 2. 流式输出（第 5 步）
- **方式**：SSE (Server-Sent Events)
- **频率**：每接收到一个 token 就推送一次
- **优势**：用户实时看到生成过程

### 3. 字数检查（第 6 步）
- **时机**：每一轮 LLM 调用结束后
- **判断**：
  - 如果 `current_words >= required_words` → 停止
  - 如果 `current_words < required_words` → 续写
  - 如果 `segment_index >= max_segments` → 强制停止

### 4. 自动续写（第 7 步）
- **触发条件**：字数不足 + 未达到最大段数
- **关键**：续写提示包含上文最后 150 字
- **保护**：明确告知 AI 不要重复

### 5. 智能滚动（第 10 步）
- **自动滚动**：用户在底部时
- **停止滚动**：用户向上滚动时
- **恢复滚动**：用户点击"回到底部"按钮

## 数据流

### 请求数据流
```
前端 → 后端 API → 数据库（保存用户消息）→ LLM API → 流式响应 → 前端
```

### 响应数据流
```
LLM API → 后端（接收 token）→ SSE 推送 → 前端（实时显示）
                ↓
         字数检查 → 续写判断
                ↓
         （如需续写）再次调用 LLM API
                ↓
         完成 → 保存到数据库 → 推送完成事件 → 前端
```

## 时间线示例

假设用户要求写 5000 字：

```
00:00  用户点击发送

00:02  后端提取字数要求：5000 字
00:03  开始第一轮 LLM 调用
00:04  前端开始接收 token，实时显示
00:05  ...持续接收...
00:30  第一轮结束，输出 3500 字
00:31  字数检查：3500 < 5000，需要续写
00:32  生成续写提示，开始第二轮 LLM 调用
00:33  前端继续接收 token
00:34  ...持续接收...
00:50  第二轮结束，总计 5100 字
00:51  字数检查：5100 >= 5000，满足要求
00:52  保存到数据库，推送完成事件
00:53  前端显示完成状态
```

总耗时：约 53 秒

## 与自动生文的区别

### AI 对话（当前流程）
- **触发**：用户在聊天框输入
- **交互**：实时对话，可以随时调整
- **字数控制**：自动提取 + 智能续写
- **输出**：保存在 `chat_messages` 表
- **用途**：快速生成、迭代修改、探索想法

### 自动生文（Celery 任务）
- **触发**：用户在任务列表提交批量任务
- **交互**：提交后后台执行，不可中断
- **字数控制**：预设的段落字数（引子 200 字、免费 3000 字等）
- **输出**：保存在 `tasks` 和 `segments` 表
- **用途**：批量生产、标准化流程、情感故事
