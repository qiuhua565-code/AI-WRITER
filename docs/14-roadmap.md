# 14 实现范围与开发顺序说明

本文档只说明**系统要交付什么**、以及**建议的代码依赖顺序**，便于前后端分工与联调；**不写周计划、不写阶段版本、不写“暂时不做”的功能清单**——具体排期由项目自行安排。

## 14.1 系统要交付的能力（与 01～13 章一致）

以下能力即本设计文档的**完整目标范围**，实现时以各章技术设计为准：

- 用户认证、个人设置、**每用户自配 LLM Key**（加密存储）
- 任务：创建（含批量）、列表、详情、状态机全路径（含暂停/继续/取消/失败重试）
- 后台编排：**按章生成**、续写、模型降级、Token 与事件记录
- **流式输出**：Redis Stream + SSE，前端流畅展示
- 可选**大纲审核**开关；审核期**段级编辑**、AI 辅助（选中/对话）、版本历史、一致性检查、Word 导出
- 任务调度：**公平用户队列**、用户级并发、容量与超时策略
- 部署、监控、告警、数据模型与 API 见对应章节

若某条与业务有出入，以**需求变更**为准改相应章节，不要在本文件里维护“第二套范围”。

## 14.2 建议的实现顺序（技术依赖，非排期）

按依赖关系从底向上，减少返工；**不是时间线**，各块可多人并行。

1. **数据与基础设施**  
   PostgreSQL 表结构、迁移、Redis、环境变量、LLM Client 单测连通中转站。

2. **认证与用户**  
   注册/登录/JWT、用户 Key 的保存与解密、鉴权中间件。

3. **任务与编排核心**  
   任务 CRUD、Celery worker、StoryOrchestrator（大纲 → 逐章 → 组装）、**续写与降级**（与 04、08 一致）。

4. **流式与任务控制**  
   Redis Stream、SSE、`pause`/`resume`/`cancel` 控制信号与 worker 协作式检查。

5. **调度与公平队列**  
   用户级轮询入队、单用户并发限流（与 07 一致）。

6. **REST API 补全**  
   按 [10-api-spec.md](./10-api-spec.md) 对齐端点。

7. **前端**  
   任务列表、批量提交、详情与流式订阅、TipTap 编辑与审核侧能力（与 11 一致）。

8. **运维与监控**  
   docker-compose、健康检查、日志、告警与定时任务（与 12、13 一致）。

## 14.3 联调检查点（验收时自测用）

用于自测是否“能力闭环”，**不作为发布阶段划分**：

- 单用户：提交任务 → 流式可见 → 生成结束进入审核 → 编辑/导出  
- 多用户：同时多任务，公平调度与单用户 key 限流符合预期  
- 异常：限流/超时/降级有记录，任务可恢复或明确失败原因  
- 暂停/继续后状态与内容一致  

## 14.4 后续增强（可选，另开需求再改设计）

如 RAG、企业 SSO、多导出格式等，**不在本文维护清单**；需要时单独补充设计章节，避免与主文档两套说法。
